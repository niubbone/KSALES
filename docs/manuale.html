1. Panoramica

Il sistema automatizza la riconciliazione delle provvigioni WKI confrontando:

  • RENDICONTI SAP (CSV mensili per linea): ciò che WKI ha effettivamente liquidato
  • PRODOTTI: le aliquote provvigionali attese per ogni prodotto/linea/tipo
  • CONTRATTI: l'anagrafica degli ordini firmati

Il risultato è il foglio KS-CS con una tab per linea, una riga per contratto, e uno STATO che indica a colpo d'occhio se la situazione è ok o richiede attenzione. Ogni linea ha anche una tab PAG con il pivot mensile dei pagamenti.

Regola GESTITE (G020, G174, X136, F502): dal 01/01/2024 non esistono rinnovi sulle gestite — tutti i contratti sono NUOVO indipendentemente da cosa risulta in CONTRATTI.tipo. La distinzione NUOVO/RINNOVO in ALIQUOTE_STORICHE per le gestite si applica solo a eventuali contratti pre-2024 marcati manualmente come RINNOVO dall'utente. Il sistema forza automaticamente tipo=NUOVO per qualsiasi contratto gestite con data_firma ≥ 2024.
Regola triennali/biennali (multi-anno con più dvend sullo stesso NC): tutte le annualità (G/GF/NT) dentro lo stesso num_contratto usano il tipo del contratto principale. Per le gestite 2024+ questo è sempre NUOVO. La provvigione si calcola sul totale — es. triennale 2700€ GESTITE = 20% su 2700 = 540€.
S148: aliq_s148 rappresenta il totale annuo (firma + incasso). La firma triennale viene pagata anno per anno. Solo contratti NUOVI — nessun rinnovo in S148.

4. Flusso operativo standard

Setup iniziale (una tantum)

  1. Eseguire setupDB → crea le tab nel DB
  2. Eseguire populateProdotti → popola il catalogo prodotti
  3. Eseguire _impostaCS (con ID reale del foglio CS) → salva l'ID nelle proprietà dello script
  4. Verificare gli ID cartelle Drive in flusso.gs (FOLDER_REND, FOLDER_F502, FOLDER_ESTR)

Import mensile — flusso automatizzato (3 fasi)

Caricare prima i file nelle cartelle Drive, poi eseguire in sequenza dal menu GAS. Ogni fase è idempotente: se va in timeout rilanciarla.

Fase 1 — Import
flusso_1_import
S148 + F502 + rendiconti G174/X136/X200 + estrazioni BudAnal dalla cartella Drive.
Fase 2 — Popola
flusso_2_popola
Nuovi clienti e contratti, backfill date firma e num_contratto, aggiorna stati.
Fase 3 — Genera
flusso_3_genera
Rigenera tutti i tab ORD e PAG nel foglio KS-CS, svuota la cache GAS. 
Frontend → "Aggiorna tutto"   → svuota localStorage e ricarica i dati

ID cartelle Drive configurati in flusso.gs:

  • Rendiconti / S148: 1ZUkyA3OnvJWGGb7UMxJk3RQI193K0ObB
  • F502: 1TX2cZK2lNcXsDCuoi-eI-H_EDf2SM-nz
  • Estrazioni BudAnal: 17MCZCjnag9YM479LGY_Fe-_HzYibDAvn
Per importare un nuovo anno di rendiconti aggiungere in importRendiconto.gs:
function importaRendicontiNNNN() { _importaAnno('NNNN'); }
e aggiornare la chiamata in flusso_1_import().

Diagnostica aggregazione contratti

Dopo flusso_2_popola, per verificare l'aggregazione:

_diagnosticRendiconti   → crea sheet DIAGNOSTIC + output Logger compatto

Il Logger (Visualizza → Log in GAS) mostra per ogni ordine non risolto: prefisso, imponibile, co-occorrenze. La sheet DIAGNOSTIC mostra: distribuzione durate, prefissi, ordini non risolti, gruppi aggregati.

Ricalcolo aggregazione da zero (dopo cambio logica backfill)

_resetNumContratto         → (opzionale) pulisce nc numerici errati
_resetDurataContratti      → solo se è cambiata _inferTipoEDurata
_backfillNumContratto      → ricalcola tutto da zero (idempotente)
_resetNumContratto è ora opzionale: il backfill azzera autonomamente i SYN_ ad ogni esecuzione.

Modifica manuale codici agenzia

  1. Editare codice_agenzia nella tab CLIENTI di KS-DB
  2. Eseguire _sincronizzaAgenzieDaClienti → propaga a CONTRATTI (con guard linea-gruppo)
  3. Eseguire flusso_3_genera → aggiorna il CS

Contratti split (triennale → annuale + biennale)

SAP a volte scompone un contratto triennale in due ordini con lo stesso N. doc. SAP. In PAG vengono automaticamente raggruppati in una riga unica con NUM_ORDINE = "ORD1 / ORD2" e DURATA_MESI = 36.

Chiave di raggruppamento: num_contratto + cliente + mese_firma.

8. DELTA e STATO — Guida dettagliata

Come viene calcolato DELTA

Il confronto avviene sulle aliquote (tassi %), non sugli importi assoluti.

tassoFirma    = tasso_rem SAP della prima riga YASV/YPFA del prodotto principale
tassoIncasso  = tasso_rem SAP della prima riga KRSV/KSBF del prodotto principale
tassoTot      = tassoFirma + tassoIncasso

aliqAttesa    = aliquota totale da PRODOTTI per (prodotto, tipo, linea)

DELTA         = tassoTot − aliqAttesa         [punti percentuale]
DELTA_PCT     = DELTA / aliqAttesa × 100      [% di scostamento]

Classificazione STATO

15. ALIQUOTE_STORICHE — gestione cambi tariffe

Permette di calcolare la provvigione corretta in base alle aliquote vigenti al momento della firma del contratto, non a quelle attuali.

Logica di lookup

_lookupAliquotaStorica(codMateriale, dataFirma):
  cerca in ALIQUOTE_STORICHE la riga dove:
    cod_materiale = codMateriale
    data_inizio ≤ dataFirma
    data_fine ≥ dataFirma  oppure  data_fine è vuota
  → restituisce le aliquote di quella riga

Usata in: apiGetContrattoDetail(), generaCS, generaStorico.

Procedura cambio tariffe (es. 01/01/2027)

  1. In ALIQUOTE_STORICHE, impostare data_fine = 31/12/2026 su tutte le righe con data_fine vuota
  2. Aggiornare PRODOTTI con le nuove aliquote
  3. Copiare le nuove righe di PRODOTTI in ALIQUOTE_STORICHE con data_inizio = 01/01/2027 e data_fine vuota
  4. Eseguire flusso_3_genera

16. Aggregazione contratti — backfillNumContratto

Problema

SAP divide un contratto triennale in più ordini: G\d (anno 1) + NT (anni 2-3) + eventualmente GF (sostituto di NT cancellato). Il sistema assegna un num_contratto comune (SYN_xxx per contratti ricostruiti, n_doc reale dove disponibile).

Architettura PASS

Strategia 2YASV+ stesso mese/cliente (guardia ratio 0.15)Ordini co-firmati; esclude satellite piccoli
PASS 1nDocMap → nc reale; azzera SYN_ precedentiRende il backfill idempotente
PASS 2Prossimità date ≤45gg + coOccMapFallback per ordini vicini
PASS 3aNUOVO → NT/GF via coOccMap; inherit-taken-ncContratti multi-prodotto (KLEOS + HYPERCARE co-firmati)
PASS 3bNUOVO con nc reale → propaga agli NT viciniPropagazione nc reale da n_doc

PASS 4GF + NT cancellato (imp≤0) stesso mese/clienteGF eredita nc dell'NT sostituito
PASS 5GK + stornato (imp<0) stesso cliente + |imp|±1€ + ≤90ggGK sostituto per ordini stornati (solo GK, non GX/GL)

Dvend ANOMALIA — significato e visibilità nel dettaglio commerciale

stato = ANOMALIA in CONTRATTI è un indicatore di anomalia provvigionale calcolato da aggiornaStatiContratti: saldo negativo + nessun incasso da >6 mesi + firma >3 mesi fa. Non indica un duplicato SAP e non esclude il dvend dal dettaglio commerciale.

I dvend ANOMALIA compaiono regolarmente in apiGetContrattoCommerciale: sono visibili nella lista ORDINI SAP, contribuiscono a imponibili e composizione, e partecipano al calcolo di provvAttesa e provvRicevuta. Lo stato ANOMALIA serve solo per il filtro nella lista contratti e per le segnalazioni.

Nota storica (@108): in precedenza apiGetContrattoCommerciale escludeva i dvend ANOMALIA confondendoli con duplicati SAP. Questo causava la sparizione dal dettaglio commerciale di dvend legittimi (es. anno 3 di un triennale X200 non ancora incassato). Il filtro è stato rimosso: gli NT stornati sono già esclusi dall'imponibile via netImpByDvend > 0.

17. Web frontend

Single-page app Alpine.js servita da GitHub Pages. Legge i dati tramite apiGetContrattoDetail esposta dal GAS Web App.

Deployment GAS Web App

clasp deploy --deploymentId {DEPLOYMENT_ID} --description "vN"
Non creare mai nuovi deployment senza --deploymentId — creerebbe un nuovo URL e romperebbe il frontend.

Cache (due livelli)

LivelloTTLCosa contiene
CacheService GAS30 mindet_xxx (dettaglio ordine), comm_xxx (contratto commerciale)
localStorage browser15 minks_getContratti, ks_getClienti

Entrambi svuotati da «Aggiorna tutto» nel frontend, oppure clearAllCache() da GAS (incluso in flusso_3_genera).

Dopo ogni deploy GAS: eseguire «Aggiorna tutto» nel frontend per svuotare anche la cache CacheService — il deploy non la invalida automaticamente.
